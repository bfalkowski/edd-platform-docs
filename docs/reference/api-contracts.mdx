The product repo includes a FastAPI platform API and generated OpenAPI contract. The public docs should stay aligned with that API, but the generated contract itself remains in the product repo.

## Contract principles

API contracts should preserve both the product loop and the object spine:

```text
Observe -> Analyze -> Measure -> Improve -> Compare -> Gate
```

```text
AgentDesign -> AgentVersion -> Scenario -> EvalContract -> Run
  -> EvalResult / JudgeOutput -> FailurePacket -> FixProposal
  -> candidate AgentVersion -> Comparison -> GateDecision
```

## Recommended conventions

- Use stable IDs for all workflow entities.
- Store evidence references separately from evidence summaries.
- Keep external evidence references typed, such as `langfuse_trace_id` or `langfuse_score_id`.
- Make comparisons explicit about baseline version, candidate version, scenarios, and eval contracts.
- Make gate decisions immutable or auditable after creation.
- Keep Sentiment Observer seeded demo objects ordinary platform records, not hardcoded special cases.

## Product repo references

In the product repo:

- `docs/API_CONTRACT.md` explains the generated OpenAPI contract and contract-first rules.
- `docs/openapi.json` is the generated OpenAPI document.
- `apps/api/edd_platform_api/main.py` contains the FastAPI routes.
- `scripts/export_openapi.py` regenerates the OpenAPI artifact.
- `scripts/lint_openapi.py` validates expected contract coverage.
